     * POST /api/auth/logout — destroys session + clears staff_sessions in Supabase

     * POST /api/staff — create staff member (master password required)
     * PATCH /api/staff/:id — update staff (master password required)
     * DELETE /api/staff/:id/sessions — kick staff (master password required)
     * GET /api/access-levels — list access levels (master password required)
     * POST /api/access-levels — create access level (master password required)
     * PATCH /api/access-levels/:id — update access level (master password required)
     * DELETE /api/access-levels/:id — delete if no staff assigned (master password required)
     * GET /api/staff/active-sessions — active sessions with last_seen_at (master password required)

─── Known limitations ────────────────────────────────────────

- Sessions stored in MemoryStore (in-process memory). Sessions will
  be lost on server restart. For production, consider using a
  persistent session store backed by Supabase/Redis.
- The master password gate in Settings is a UX gate only — it is
  checked client-side. API routes are protected by the x-master-password
  header server-side, which is the actual security boundary.
